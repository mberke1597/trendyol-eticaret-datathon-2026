You are an expert Data Scientist and Machine Learning Engineer specializing in E-Commerce Search, Information Retrieval (IR), and Learning to Rank (LTR). I want you to design a state-of-the-art solution for a query-product relevance ranking competition.

Here are the complete details of the competition:

### 1. Competition Task & Goal
The objective is to predict whether a given product (item) is relevant to a user search query. 
- The target is binary: `1` if the product was clicked by a user during a search session for that query, and `0` otherwise.
- The evaluation metric is **Macro F1-Score** on a hidden test set.

### 2. Dataset Structure
We have the following files:
1. **`terms.csv`**: Contains search queries.
   - Columns: `term_id`, `query` (Turkish e-commerce search queries, e.g., "iphone 13 kılıf", "kadın spor ayakkabı").
2. **`items.csv`**: Metadata of products.
   - Columns: `item_id`, `title` (product title), `category` (hierarchical category path, e.g., "ayakkabı/spor ayakkabı"), `brand` (brand name), `gender` (target gender: erkek, kadın, unisex, unknown), `age_group` (yetişkin, çocuk, bebek, unknown), `attributes` (JSON/text of product attributes like color, material, pattern).
3. **`training_pairs.csv`**: 250,000 clicked pairs.
   - Columns: `term_id`, `item_id`, `label` (all labels are `1` in this file).
4. **`submission_pairs.csv`**: 3,360,000 candidate pairs to predict on.
   - Columns: `id`, `term_id`, `item_id`.
   - **CRITICAL DETAIL:** The test set contains top-K candidates generated by a baseline retriever (BM25 + FAISS). Therefore, the test set consists ONLY of hard candidates (high semantic similarity/overlap), not random pairs.

### 3. Available Models & Technical Constraints
- We have precomputed dense embeddings using a domain-specific encoder (`trendyol/TY-ecomm-embed` - 768d) and a lightweight model (`Turkish TinyBERT` - 128d).
- **Execution Time Concern:** The test set has 3.3 million pairs. Inference must be computationally efficient. Heavy deep learning cross-encoders during test inference are not allowed; we must rely on precomputed embeddings and fast feature engineering (e.g., L2/cosine similarities, token-level matching) combined with GBDTs.

### 4. Key Challenges & Domain Dynamics
You must address the following e-commerce search challenges:
- **Negative Sampling & Calibration:** The original training file only contains positive clicks (`label=1`). We must generate synthetic negatives (`label=0`). Since the test set only contains retrieved candidates, random negative sampling might create a domain shift (easy negatives vs. hard test candidates).
- **Linguistic Variations (Turkish NLP):** Turkish is an agglutinative language. Suffixes (e.g., "okuma kitabı" vs. "okuma kitabını"), spacing mismatches (e.g., "multi vitamin" vs. "multivitamin"), spelling typos, and brand abbreviations are highly common.
- **Noisy Seller Metadata & Hard Constraints:** Product metadata (gender, age group) provided by sellers is frequently incorrect or missing. However, query constraints (e.g., a search for "kadın ayakkabı" must never return a "male" shoe) are absolute. How do we design robust contradiction checkers that bypass seller metadata noise?
- **Popularity Bias vs. Relevance:** Click logs are naturally biased towards highly popular brands or categories, which can degrade relevance generalization on long-tail queries. How do we prevent the model from overfitting to popularity clicks?

### 5. Raw Signals Inventory (Available Building Blocks for Feature Engineering)
To help you brainstorm features, here are the exact raw signals available in our database for each query-product pair:
- **Text Fields:** Raw query, raw product title, hierarchical category path (slash-separated string), brand string, raw attributes (comma-separated key-value text).
- **Categorical Meta:** Target gender (categorical), target age group (categorical).
- **Historical Popularity Stats:** We can compute historical click counts for brands, categories, or items from the training log (but beware of popularity bias).
- **Embeddings:** 768-dimensional dense vectors for queries, titles, and categories, and 128-dimensional dense vectors for query and product attributes.
